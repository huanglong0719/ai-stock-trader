**实施日期**: 2026-01-23

### 1. 分钟线下载与落库（5min/30min）
- **目标**: 支持沪深股票分钟线批量下载，盘后增量落库，前端展示稳定且快速。
- **实现要点**:

  - 落库表统一为 `minute_bars`，唯一键 `(ts_code, trade_time, freq)`，避免重复写入与历史回补冲突。
  - 同步写入 Redis 最新 800 条，供盘中快速查询与“最新数据补丁”。

### 2. 盘后自动增量同步（16:00 后）
- **背景**: 通达信盘后数据通常需要 16:00 之后才稳定可用。
- **实现要点**:
  - 新增调度任务 `post_close_minute_sync`（默认 16:10，可在 `.env` 配置），盘后自动增量同步分钟线。
  - 可配置项：`MINUTE_AUTO_SYNC_AFTER_CLOSE / MINUTE_SYNC_POOL / MINUTE_SYNC_DAYS / MINUTE_SYNC_FREQS / MINUTE_SYNC_LIMIT / MINUTE_SYNC_CONCURRENCY / MINUTE_SYNC_AFTER_CLOSE_HOUR / MINUTE_SYNC_AFTER_CLOSE_MINUTE`。

### 3. 分钟线接口稳定性优化（减少偶发慢）
- **问题**: 分钟线偶发变慢通常来自“本地无数据 → 回退 TDX 在线拉取”，且 TDX 连接抖动会放大延迟。
- **优化策略**:
  - `/api/market/kline` 的分钟分支改为优先读取 `minute_bars`，再拼接 Redis 实时数据，最后才回退 TDX。
  - 限制分钟线返回点数上限，避免一次返回过大导致前端渲染卡顿。

### 4. 盘中任务超时治理（trade_monitor / 上下文构建）
- **问题**: 盘中 `trade_monitor` 在持仓/计划多、AI/搜索慢时可能触发 180s 超时。
- **优化策略**:
  - 对计划处理、持仓处理、上下文批量获取加入并发上限与任务级超时，避免无限并发造成雪崩。
  - 为搜索/盘口/板块等慢调用加超时降级，保证任务“可完成”优先。

## �🔴 2026-01-20 资金对账口径统一与资产同步并发覆盖修复 (Cash Reconciliation & Sync Safety)
**实施日期**: 2026-01-20

### 1. 资金对账口径统一（TradeRecord.amount / fee）
- **问题背景**: 用户要求用“今日所有买卖明细”手工核对资金曲线，发现可用资金变化与成交现金流不一致。
- **统一口径**:
  - BUY：`TradeRecord.amount = 成交金额(不含手续费)`，`TradeRecord.fee = 买入费用`；现金流出应按 `amount + fee`。
  - SELL：`TradeRecord.amount = 净入账金额(net_amount)`，`TradeRecord.fee = 卖出总费用`；现金流入直接用 `amount`，不应再扣一次 `fee`。
- **落地位置**:
  - 成交写入：`execute_buy / execute_sell`（交易记录字段写入与资金加减一致）
  - 审计脚本：`detail_audit.py` 按上述口径计算现金流

### 2. 账户现金对账（以成交记录为唯一现金流水来源）
- **目标**: 当账户现金出现偏差时，能用成交记录与冻结资金快速恢复一致性，确保资金曲线可复现、可审计。
- **实现逻辑**:
  - 预期总现金：`initial_capital - Σ(BUY.amount+BUY.fee) + Σ(SELL.amount)`
  - 预期冻结：`Σ(trading_plans.frozen_amount)`（仅统计 >0）
  - 预期可用：`expected_total_cash - expected_frozen`
- **落地位置**: `trading_service.py` 的 `_calc_expected_total_cash / _calc_expected_frozen_cash / reconcile_account_cash`

### 3. 资产同步并发安全补强（避免旧对象覆盖新数据）
- **风险场景**: `sync_account_assets` 读取到的 `account` 可能在锁外被其它交易/解冻流程更新，若不刷新对象，在再次持锁写入时可能把其它线程刚写入的数据覆盖回旧值。
- **修复要点**:
  - 现金对账后在锁内立即 `commit + refresh`，确保对象进入“最新状态”。
  - 市值写入前在锁内再次 `refresh(account)`，再更新 `market_value / total_assets`，避免覆盖并发写入的 `available_cash / frozen_cash`。
- **落地位置**: `trading_service.py` 的 `sync_account_assets`

## 🔴 2026-01-16 账户资产偏差修复与并发竞争治理 (Asset Discrepancy & Race Condition Fix)
**实施日期**: 2026-01-16

### 1. 资产虚高根因诊断

### 1. 资产虚高根因诊断
- **背景**: 用户反馈模拟账户收益率从 5.87% 异常跳升至 16.86%，且可用资金出现约 10.6 万元的虚增。
- **诊断结论**: 系统存在严重的**并发竞争条件 (Race Condition)**。
    - **场景**: 交易执行线程 (`execute_sell`) 与后台解冻任务 (`unfreeze_all_expired_plans` / `_process_single_plan`) 在毫秒级时间内并发运行。
    - **过程**: 
        1. 后台任务查询出“有冻结资金”的计划对象 P (Memory Object A)。
        2. 交易线程执行卖出，解冻资金，并将数据库中 P 的 `frozen_amount` 清零。
        3. 后台任务获取锁，但直接使用了旧对象 P (Memory Object A) 进行解冻操作。
    - **结果**: 同一笔冻结资金被交易线程和后台任务分别加回账户，导致“双重解冻”。

### 2. 并发安全治理
- **强制刷新 (State Refresh)**: 在所有涉及资金解冻的临界区（Lock Scope）内，强制执行 `db.refresh(plan)`，确保操作前获取数据库最新状态。
- **幂等性检查 (Idempotency)**: 在解冻逻辑 `_unfreeze_funds` 调用前，增加 `if plan.frozen_amount > 0 and not plan.executed` 的二次校验。
- **范围**: 覆盖了自动交易、AI 撤单、AI 观望解冻、盘后清理等所有资金释放入口。

### 3. 历史资产修正
- **数据清洗**: 开发并执行了 `fix_account_assets.py` 脚本。
    - **回溯算法**: 基于 `trade_records` 表中的 68 笔历史成交记录，从初始资金 (100万) 开始逐笔重算现金流。
    - **市值校准**: 结合最新持仓与实时行情，重新计算总市值。
- **修复结果**: 
    - 剔除约 11.6 万元虚增资金。
    - 总资产从 117.8 万修正为 106.2 万。
    - 总收益率从 16.86% 修正为 6.23% (回归真实水平)。

## 🟢 2026-01-14 模拟撮合规则对齐：涨跌停排队、申报单位、手续费 (A-Share Microstructure Alignment)
**实施日期**: 2026-01-14

### 0. 委托资金冻结/解冻与参考价回退逻辑同步
- **背景**: 用户反馈存在“未冻结的买入计划”，尤其是市价类或策略名不在白名单的计划，导致委托列表出现“未冻结”的风险提示。
- **问题根因**:
  - 旧版 `_freeze_funds` 通过策略名白名单与 `buy_price_limit>0` 双条件判断买入意图，导致市价买入或新策略名不在白名单时直接跳过冻结。
  - 参考价获取仅依赖限价与实时报价，盘后或行情源短时不可用时会得到 0，从而放弃冻结。
- **改进逻辑**:
  - 新增 `_infer_plan_action(plan)` 统一判定计划意图，优先使用 `ai_decision`，并识别卖出/观望类；默认其余视为买入。
  - `_freeze_funds` 与 `_realign_frozen_funds` 改为基于 `_infer_plan_action(plan) == 'BUY'` 且 `position_pct>0` 的统一入口；移除旧版策略白名单判断。
  - 参考价获取顺序增强：`buy_price_limit / limit_price / decision_price → 实时报价 → 最新日K收盘价 (DailyBar.close)`；严格模式下若仍为 0，抛出 400 以便任务侧跳过该标的。
- **效果**:
  - 市价买入、AI 生成的新策略名买入、盘后生成买入计划在计划创建/更新或挂单阶段均能正确冻结。
  - 冻结失败原因在严格模式下清晰可见（返回 400），避免静默遗漏。
  - 与前端“委托详情”的状态审计一致：BUY 未冻结时给出明确的“未冻结：预计需X，可用Y”的提示。

### 1. 涨跌停排队模拟 (Limit-up / Limit-down Queue)
- **背景**: A 股在涨停/跌停封死时，委托通常进入排队；模拟交易需要体现“未轮到不能成交”的制度差异。
- **实施逻辑**:
    - 在 `trading_service.py` 中实现 `_check_limit_order_queue(db, plan, quote, side, order_vol)`，并以 `_shares_to_hands` 统一委托量口径（股 -> 手）。
    - **触发条件**:
        - 买入：`side="BUY"` 且 `current_price >= limit_up`。
        - 卖出：`side="SELL"` 且 `current_price <= limit_down`。
    - **直接成交判定**:
        - 涨停买入：若卖一量 `s1_v > 0` 且 `s1_v >= order_hands`，认为有对手盘可直接成交，并清理 `queue_info`。
        - 跌停卖出：若买一量 `b1_v > 0` 且 `b1_v >= order_hands`，认为有对手盘可直接成交，并清理 `queue_info`。
    - **进入排队与快照记录**:
        - 若封死且无法直接成交，则在 `plan.market_snapshot_json` 写入 `queue_info`：
            - `queue_type`: `LIMIT_UP_BUY` / `LIMIT_DOWN_SELL`
            - `start_vol`: 进入排队时的 `quote.vol`（手）
            - `ahead_hands`: 涨停买入记录 `b1_v`（手），跌停卖出记录 `s1_v`（手）
            - `order_hands`: 当前委托量（手）
            - `start_time`: 排队开始时间
        - 同步写入 `plan.review_content`，用于前端展示“开始排队/等待排队”的可读提示。
    - **出队判定**:
        - `delta_vol = quote.vol - queue_info.start_vol`
        - `need_clear = queue_info.ahead_hands + queue_info.order_hands`
        - 当 `delta_vol >= need_clear` 且 `need_clear > 0` 时，视为队列被成交量“吞掉”，清理 `queue_info` 并允许成交。
- **效果**: 涨跌停场景不再“瞬间成交”，系统能稳定模拟排队等待与队列清空后的成交窗口。

### 2. 申报数量规则统一 (Order Size Rules)
- **背景**: 不同板块存在最小申报与步长差异，若“冻结/计算/下单”口径不一致，会出现委托金额冻结成功但成交端拒绝的异常体验。
- **实施逻辑**:
    - 在 `trading_service.py` 中统一三件事：最小申报、步长、归一化。
        - `_get_min_buy_volume(ts_code)`: 默认 100 股；科创板（`688*`）为 200 股。
        - `_get_buy_volume_step(ts_code)`: 默认 100 股；科创板（`688*`）步长为 1 股。
        - `_normalize_buy_volume(ts_code, volume)`: 按步长向下取整；步长为 1 时不做截断。
    - `execute_buy` 在计算出 `volume` 后统一执行归一化与最小申报校验，并在冻结资金逻辑中复用同一套口径。
- **效果**: 科创板委托数量规则与主板规则可共存，且资金冻结、委托计算、成交校验三处行为一致。

### 3. 手续费与过户费口径补齐 (Fee Model Completion)
- **背景**: 模拟交易若缺少卖出印花税与沪市过户费，会造成盈亏与资金变化长期偏离真实交易口径。
- **实施逻辑**:
    - 买入佣金 `_calc_buy_fee(amount)`: `max(5元, amount * 0.00025)`。
    - 过户费 `_calc_transfer_fee(ts_code, amount)`: 仅沪市（`.SH`）收取 `amount * 0.00001`。
    - 卖出费用 `_calc_sell_fees(ts_code, amount)`: `佣金 + 印花税(amount * 0.0005) + 过户费`。
    - 买入成交：`fee = 佣金 + 过户费`，计入 `TradeRecord.fee` 并影响持仓加权成本。
    - 卖出成交：`net_amount = amount - total_fee` 计入可用资金，成交记录同步写入费用。
- **效果**: 资金与盈亏口径更贴近 A 股真实成交回报，且在买卖两侧费用拆分明确。

### 4. 交易时间与规则严格对齐 (Trading Time & Rules Alignment)
- **背景**: 系统原先允许全天候挂单，违反 A 股实际交易规则（如集合竞价和午休时间不可成交）。
- **实施逻辑**:
    - **挂单拦截**: 在 `execute_buy` 和 `execute_sell` 入口处增加 `is_trading_time()` 校验，非交易时间直接拒绝并记录警告。
    - **集合竞价限制**: 增加对 `9:15-9:25` 时间段的判断，此期间挂单将被系统拦截（返回 False），模拟交易所未开盘状态。
    - **午间休市限制**: 增加对 `11:30-13:00` 时间段的判断，此期间挂单将被系统拦截。
- **效果**: 系统严格遵守 A 股时间规则，杜绝了非交易时间产生无效成交或挂单的异常情况。

## 🟢 2026-01-13 自动化资金管理流与“废单”状态逻辑 (Automated Fund Management & Invalid Orders)
**实施日期**: 2026-01-13

### 1. 挂单与资金冻结全生命周期 (Fund Life-cycle)
- **背景**: 用户需要完整的资金管理流程：挂单冻结 -> 成交释放 -> 自动解冻。
- **实施逻辑**: 
    - **下单冻结**: 对“今日买入类计划”，在计划创建/更新时即通过 `_freeze_funds` 尝试冻结（按 `总资产 * 仓位比例` 生成预算，并结合参考价计算可申报数量与预估手续费后写入 `frozen_amount/frozen_vol`）；若 AI 后续给出“挂单但暂不成交”的决策且计划仍未冻结，则在挂单阶段补充冻结。
    - **资金不足拦截**: 对买入类计划启用严格冻结模式，若可用资金不足以满足最小委托（含费），则该委托不会发出（对外返回 400；系统任务侧跳过该股票并继续处理其他标的）。
    - **成交释放**: 成交回报处理时，自动调用解冻逻辑，将 `frozen_amount` 转化为持仓市值或返还余数。
    - **收盘自动解冻**: 在 `scheduler.py` 中新增 `15:05` 定时任务 `unfreeze_funds_job`。
        - 逻辑：扫描今日所有未执行且有冻结资金的计划。
        - 判定：如果已过 15:00 且未成交，强制执行 `_unfreeze_funds`，并将资金归还至可用余额。
    - **AI 撤单解冻**: 针对 AI 决策执行 `CANCEL` 的情况，在 `_process_single_plan` 中立即触发资金解冻，确保资金周转效率。
- **效果**: 实现了模拟交易账户的资金闭环管理，解决了挂单导致资金“消失”或无法复用的问题。

### 2. “废单”判定与计划解耦 (Invalid Order Logic)
- **背景**: 用户明确要求“只有委托里面才有废单，与计划无关”。
- **实施逻辑**: 
    - **存储一致性**: 数据库 `trading_plans` 表不再记录临时的“废单”状态，以保持交易计划作为“原始指令”的纯净性。
    - **接口动态计算**: 修改 `trading_endpoints.py` 中的 `get_entrustments` 接口。
        - 判定逻辑：如果一个委托记录（TradingPlan）满足 `not executed` 且 `date < today` 或 `is_trading_time() == False`，则在返回前端时动态映射为 `status = "废单"`。
    - **前端视觉区分**: 修改 `TradingPanel.jsx`，为 `废单` 状态配置 `error` (红色) 标签；委托理由默认省略显示，悬停可查看完整内容。
- **效果**: 实现了逻辑上的解耦。用户在“今日计划”中看到的是原始决策，在“委托详情”中看到的是实际执行状态（包含已失效的废单）。

### 3. 账户资产计算一致性修复 (Asset Consistency)
- **问题诊断**: 发现 `Total != Available + Frozen + MarketValue` 的数据偏差。
- **修复逻辑**: 
    - 编写并执行了数据校准脚本，重新计算所有历史持仓的当前市值。
    - 统一了资产计算公式：`total_assets = available_cash + frozen_amount + total_market_value`。
- **效果**: 确保了账户大屏显示的数据 100% 符合财务逻辑。

## 🟢 2026-01-13 多频率指标存储修复与 K 线加载性能优化 (Multi-frequency Indicators & Performance)
**实施日期**: 2026-01-13

### 1. 周线/月线指标存储隔离修复
- **背景**: 发现 600865.SH 等股票点击月线显示失败，或周/月线指标数值不正确。
- **问题诊断**: `MarketDataService._background_save_kline` 在持久化 API 返回的 K 线数据时，未区分频率，统一将指标存入了日线列（如 `ma5`, `macd`），导致查询 `weekly_ma5` 等列时数据为空。
- **实施逻辑**: 
    - 修改 `market_data_service.py`，根据 `freq` 参数动态映射存储字段。
    - `freq='W'` 映射至 `weekly_*` 列，`freq='M'` 映射至 `monthly_*` 列。
- **效果**: 解决了多频率 K 线指标显示不全的问题，确保了数据库存储的频率一致性。

### 2. K 线获取性能深度优化
- **背景**: 首次加载未缓存股票需 2-3 秒，用户体验欠佳。
- **实施逻辑**: 
    - **二级内存缓存**: 在 `MarketDataService` 内部引入基于 `ts_code + freq + start + end` 的 30 秒内存缓存，避免前端组件频繁挂载导致的重复数据库查询。
    - **宽容的本地数据判定**: 优化了 `is_sufficient` 逻辑。如果本地数据最新日期与当前交易日仅差 1-2 天（通常为盘中或未同步），则判定为“足够”，并直接通过 `merge_realtime_to_kline` 合并实时行情，不再触发耗时的 API 请求。
- **效果**: 未缓存股票首次加载速度从 2s+ 降低至 0.5s 以内，极大提升了流畅度。

## 🟢 2026-01-12 指标计算存储架构重构与数据修复 (Indicator Architecture & Data Repair)
**实施日期**: 2026-01-12

### 1. 均线与 MACD 异常根因修复
- **背景**: 发现 301282.SZ 等股票在 2025-04 期间出现均线异常及 MACD 缺失。
- **问题诊断**: 
    1. **复权因子关联错误**: `IndicatorService` 在存储指标时关联了“计算时的最新复权因子”而非“指标所属交易日的实际复权因子”。当复权因子随时间变动时，历史指标的缩放锚点失效。
    2. **非原始值存储**: 指标以“复权后的值”存储在数据库中，导致在不同基准日期下读取时，无法通过简单的比例换算还原，造成数据“跳水”。
    3. **缓存失效逻辑不严密**: `_filter_existing` 未检查关键指标（如 MACD）是否为空，导致部分计算失败的空值被永久缓存。
- **实施逻辑**: 
    1. **存储架构调整**: 修改 `indicator_service.py`，所有技术指标在入库前通过当天的 `adj_factor` 还原为 **原始值 (Unadjusted)**。
    2. **关联当日因子**: 存储记录时，`adj_factor` 字段严格对应交易日当天的复权因子。
    3. **动态前复权转换**: 修改 `market_data_service.py` 的 `_apply_qfq`，前端请求时统一按公式转换：$Price_{qfq} = Price_{raw} \times \frac{adj\_factor_{day}}{adj\_factor_{latest}}$。
    4. **强制空值重算**: 优化过滤逻辑，检测到数据库中关键指标（MACD）为 `NULL` 时，自动触发重算。
- **修复效果**: 彻底解决了复权变动导致的指标“断层”和“漂移”问题。经验证，301282.SZ 及 000608.SZ 的均线与 MACD 已恢复正常。

### 2. 遗留问题记录：历史数据长度不足导致指标缺失
- **现状**: 部分新上市股票（如 001335.SZ）或本地数据断层严重的股票，其日线 K 线数量不足（如少于 30 根），导致 MACD 等长周期指标无法算出，数据库中显示为 `NULL`。
- **后续计划**: 
    1. **数据回溯**: 在同步脚本中增加对新股上市首日至今的强制补全逻辑。
    2. **指标容错**: 在前端或 AI 上下文中，针对历史不足 150 天的标的给出“数据不足”提示，而非直接报错。
    3. **长周期补齐**: 定期扫描数据库中 `MACD` 为空的非指数记录，尝试从 API 重新抓取更长的历史背景数据进行补算。

## 🟠 2026-01-12 均线异常诊断与复权逻辑加固 (MA & QFQ Robustness)

- **改进逻辑**:
    1.  **指标计算 QFQ 化**: 修改 `IndicatorService`，在计算指标前先对 K 线进行前复权 (QFQ)，确保指标基于“除权后”的连续价格计算。

        - 修改 `MarketDataService._apply_qfq`，对指标字段使用其自带的 `ind_adj_factor` 进行比例换算：$Ratio = \frac{ind\_adj\_factor}{current\_latest\_adj}$。
- **效果**: 彻底解决了指标在不同复权阶段可能出现的漂移问题，确保 AI 看到的均线与当前价格在同一参考系下。

### 5. T+1 交易制度严格化 (2026-01-12)

- **效果**: 系统严格遵循 T+1 制度，当日买入份额在持仓列表中显示可用为 0，次日开盘前自动解锁。

### 6. 账户资产同步与结算严密化 (2026-01-12)

- **原因**: 调用 `/api/trading/review/daily` 时出现 `Pydantic ValidationError`。原因是 `review_service` 生成的 `target_plan` 字典中使用了 `strategy_name` 字段，而 API 定义的 `TargetPlan` 模型期待 `strategy` 字段。
- **修复逻辑**:
    - **字段对齐**: 修改 `review_service.py` 中的 `_generate_ai_plan` 方法，将返回字典中的 `strategy_name` 统一修改为 `strategy`。
    - **模型扩展**: 更新 `trading_endpoints.py` 中的 `TargetPlan` 模型，增加 `score`, `target_price`, `action` 等可选字段，以增强对 AI 生成数据的兼容性。
    - **默认值补全**: 在 `_generate_ai_plan` 中增加 `position_pct` 默认值，确保满足模型验证要求。
- **效果**: 彻底解决了每日复盘生成后的序列化失败问题，接口恢复正常响应。
