## 0. 数据生命线 (Data Lifeline - 2026-01-08 新增)

1.  **5年生命窗口**: 系统所有 K 线历史数据强制限制在 **最近 5 年**。超过 5 年的数据视为无效冗余，严禁进入生产库或 AI 分析上下文。
2.  **单一事实来源 (Single Source of Truth)**: 
    *   所有 AI 分析模块必须调用 `chat_service.get_ai_trading_context` 获取数据。
    *   严禁在 `stock_selector`, `review_service`, `trading_service` 中直接调用 `data_provider` 获取 K 线。
3.  **并行与超时基准**: 
    *   AI 上下文获取必须使用 `asyncio.gather` 并行化。
    *   所有外部数据请求必须设置明确的超时时间与降级路径，避免阻塞主流程。
4.  **复权标准**: 全系统统一使用 **前复权 (QFQ)** 数据，严禁混用。
5.  **指标计算准则**:
    *   **增量优先**: 每日数据同步后，仅对新增日期进行增量指标计算。
    *   **复权一致性校验**: 必须持久化 `adj_factor` 到指标表。计算前校验当前复权因子与记录值，若不一致（发生除权除息），必须强制触发该股全量重算。
    *   **按需计算**: 盘中实时计算应仅针对系统请求的特定标的，严禁进行全量股票的实时扫描计算，以保护 CPU 资源。
    *   **特殊标的处理**: 指数与行业板块指标计算不检查复权因子（固定为 1.0），且需支持特殊的代码格式解析（如 `IND_` 前缀）。
    *   **指标计算完整性 (2026-01-12 新增)**:
        *   **禁止伪造数据**: 严禁使用 `fillna(close)` 等逻辑填充缺失的指标（如 MA/MACD）。若数据不足以计算，必须保持为 `None`，以防误导交易决策。
        *   **计算背景要求**: 任何涉及 MA60、MACD 等中长期指标的计算，必须确保至少有 **150 个交易日** 的历史数据作为预热背景。
        *   **本地优先原则 (2026-01-13 优化)**: 系统必须优先从本地数据库获取已计算好的指标。
        *   **多频率列对齐**: 周线和月线指标必须严格映射到指标表中的 `weekly_*` 和 `monthly_*` 字段。严禁将不同频率的指标混入日线字段。
        *   **实时补齐优化**: 当本地数据仅缺失最近 1-2 天（非交易日或未大同步）时，优先通过实时行情 (Realtime Quote) 补齐，避免触发昂贵的 API 全量下载。
        *   **短期内存缓存**: `MarketDataService` 使用秒级缓存减少重复 I/O（如报价 5 秒、K 线 30 秒、板块 60 秒等）。
6.  **全市场统计精度准则 (2026-01-12 新增)**:
    *   **分页对齐**: 涉及全市场涨跌家数统计的 API 调用，必须实现自动分页抓取逻辑，严禁仅取首页数据。
    *   **北交所限定**: 北交所数据获取必须附加 `s:2048` 过滤条件，严禁混入新三板非上市品种。
    *   **阈值对齐**: 涨跌停判定必须对齐主流软件（如通达信）的四舍五入规则（9.94%/19.94%/29.94%）。

## 1. 核心原则 (Core Principles)

1.  **风险优先 (Risk First)**
    *   任何交易决策逻辑必须包含止损保护。
    *   系统异常（网络断连、API 报错）时，默认行为应为“暂停开仓”或“保持现状”，而非盲目操作。
    *   资金管理：单只股票仓位严格限制，保留充足流动资金。

2.  **数据闭环 (Data Closed Loop)**
    *   **一切皆数据**：每一次 AI 的决策（买入/卖出/观望）都必须记录当时的“输入”（市场快照、K线形态）和“输出”（决策理由）。
    *   **结果导向**：每一笔交易必须追踪最终的盈亏（Real PnL），并回填到对应的决策记录中，形成 `(Context, Action, Reward)` 的完整样本。
    *   **禁止遗忘**：成功的交易模式必须存入 `PatternLibrary`，失败的教训存入反面案例库（待建）。

3.  **真实性 (Authenticity)**
    *   严禁使用未来函数。
    *   回测与模拟必须考虑滑点、手续费（印花税+佣金）。
    *   所有开发测试必须基于真实历史数据或实时行情。

4.  *   **交易制度一致性 (A股规则对齐 - 2026-01-15 更新)**
    *   **T+1**: 当日买入份额禁止卖出；卖出校验以 `available_vol` 为准。
    *   **交易时间限制**:
        *   **挂单拦截**: 非交易时间（9:15前, 11:30-13:00, 15:00后）系统必须拒绝执行新的买入/卖出指令。
        *   **集合竞价**: 9:15-9:25 期间的挂单请求应被系统拦截（返回 False），等待 9:25 开盘价产生后重新判定。
        *   **午间休市**: 11:30-13:00 期间的挂单请求应被系统拦截。
    *   **最小申报与步长**:
        *   默认最小 100 股，且为 100 股整数倍。
        *   科创板（`688*`）最小 200 股，步长为 1 股（可按 200、201、202… 申报）。
        *   买入计算、冻结、下单校验必须使用同一口径，避免“冻结能通过但成交被拒”的不一致。
    *   **费用口径**:
        *   佣金：`max(5元, 成交额 * 0.00025)`（买卖均收）。
        *   印花税：卖出收取 `成交额 * 0.0005`。
        *   过户费：仅沪市（`.SH`）收取 `成交额 * 0.00001`（买卖均收）。
        *   买入总费用 = 佣金 + 过户费；卖出总费用 = 佣金 + 印花税 + 过户费。
    *   **涨跌停排队**:
        *   触发涨停买入或跌停卖出时，必须模拟排队；未轮到成交前返回“不可成交”并持续等待。
        *   排队进度以“盘口一档量（手） + 成交量增量（手）”驱动，并在计划快照中记录 `queue_info` 以支持可视化与复盘。
    *   **价格偏离保护**:
        *   市价买入：若 `current_price > suggested_price * 1.005`，跳过以避免追高滑点。
        *   市价卖出：若 `current_price < suggested_price * 0.995`，跳过以避免恐慌砸盘。
        *   限价单必须严格尊重计划设定的限价约束。

## 2. 架构与代码规范 (Architecture & Coding)

1.  **服务化设计 (Service-Oriented)**
    *   `data_provider`: 市场数据底层访问层（Tushare 限制、缓存、清洗）。**AI 分析上下文必须走 `chat_service.get_ai_trading_context`**。
    *   `ai_service`: 兼容门面（Facade），对外提供分析能力；底层模型调用由 `ai_client` 承担。
    *   `trading_service`: 唯一的交易执行者，负责账户状态管理、订单路由和流水记录。
    *   **禁止越级**：业务逻辑层（如 `stock_selector`）不应直接调用数据库底层 SQL，应通过 Model 操作。

2.  **状态管理 (State Management)**
    *   **服务无状态**：服务重启不应丢失关键数据。所有交易计划、持仓状态、运行日志必须持久化到 SQLite 数据库。
    *   **内存仅作缓存**：盘中高频数据（如实时价格）可短暂停留内存，但关键决策后必须立即落库。

3.  **API 交互规范**
    *   **Tushare**: 严格遵守 `TUSHARE_MAX_CALLS_PER_MINUTE` 配置的访问限制，并提供重试机制。
    *   **LLM**:
        *   必须设置超时时间（Timeout），并处理 JSON 解析失败的异常。
        *   必须做并发控制：服务层 `asyncio.Semaphore`（默认 3）+ 模型层并发门（默认每模型 2）。
        *   必须具备容错降级：优先 MiMo，遇到限流（429/RateLimit）或异常时自动降级到 DeepSeek。
        *   网络兼容性优先：默认禁用 HTTP/2，避免部分网络环境连接不稳定。

4.  **数据库异步与线程安全 (2026-01-16 新增)**
    *   **SessionLocal 线程安全**: `SessionLocal()` 实例不是线程安全的。同一个 Session 严禁被多个线程并发使用；涉及 `asyncio.to_thread` 的数据库操作必须在锁内串行化，或在 `to_thread` 内创建并关闭独立 Session。
    *   **优先写法**: 在 `to_thread` 回调内部完成 `query/commit/close`，只向外返回标量/字典等纯数据，避免跨线程传递 ORM 实例。
    *   **一致性要求**: 同一段业务流程中，冻结/解冻/资产同步等关键更新必须在同一把账户锁保护下完成，避免出现“冻结成功但资产未同步”等不一致状态。

## 3. 开发流程 (Development Workflow)

1.  **验证驱动 (Verification Driven)**
    *   修改核心交易或数据流逻辑后，必须运行 `backend/_tmp_e2e_test.py`，验证完整闭环（行情→上下文→AI 结果）。
    *   质量验证必须通过 `ruff` 与 `mypy` 检查。
    *   **分批验证逻辑**: 在涉及大数据处理（如全市场指标重算、历史数据同步）时，严禁直接全量运行。必须先处理少量数据（如 10-30 只股票）并进行结果校验，确认逻辑正确后再进行全量批处理。
    *   重大变更（如数据库 Schema 修改）必须编写独立的迁移脚本，禁止手动修改线上库。

2.  **文档化 (Documentation)**
    *   代码注释重点解释“为什么这样做”（Why），而非“做了什么”（What）。
    *   **实现逻辑记录**: 每一个核心功能或复杂逻辑的实现，必须在 `IMPLEMENTATION_LOG.md` 中详细记录其技术步骤、核心算法及处理流程，以便于后续优化与维护。
    *   每次完成大的功能模块（Phase），必须更新 `AI 炒股系统开发计划.md`。

## 5. 进化路线 (Evolution Roadmap)

1.  **L1 辅助交易** (已完成)：AI 选股，人工确认。
2.  **L2 半自动交易** (当前阶段)：AI 制定计划，自动执行买卖，人工监管。（在没有人工监管的情况下，也能够按照计划自动执行）
3.  **L3 数据闭环** (进行中)：系统自动收集胜负案例，积累经验库。
4.  **L4 自我优化** (下一阶段)：定期跑批训练，优化 Prompt 和策略权重，无需改代码即可提升胜率。

## 6. AI 决策一致性与记忆规范 (2026-01-08 新增)

1.  **分值一致性记忆 (Consistency Memory)**:
    *   **4小时缓存锁定**: 系统必须实现 `_score_consistency_cache`，记录每只标的在最近 4 小时内的 AI 评分。
    *   **跨入口对齐**: 无论是“手动点击分析”、“选股列表刷新”还是“盘中自动监控”，必须共享同一评分背景。严禁对同一标的在短时间内给出分歧巨大的结论。
    *   **分值锚定规则**: 若当前分析结论与前次评分（prev_score）偏差超过 15 分，AI 必须在分析报告中给出“极度充分”的逻辑支撑（如突发利空、放量破位），否则视为逻辑失败。

2.  **趋势决定论 (Trend Hierarchy)**:
    *   **大周期绝对优先**: 遵循“月线定性、周线定势、日线定点”的铁律。
    *   **长线偏见**: 如果月线和周线处于健康的上涨趋势，严禁因日线级别的单日波动（如放量上影线、缩量回调）给出极度悲观的评价。AI 必须学会“忽略噪音，守住趋势”。

3.  **提示词 (Prompt) 稳定性**:
    *   **统一核心准则**: 所有 AI 分析入口（Chat, API, Task）必须强制注入 `prompt_builder.get_core_analysis_rules()`。
    *   **历史背景感知**: 每次分析必须尝试获取该股的 `prev_score` 并告知 AI，赋予其“历史记忆”。
